Convenção de nomes

  • Variavéis, métodos e classes devem usar o estilo CamelCase (Java)
  • Revelam a intenção
  • Fácil de pesquisar

Não revela a sua intenção

					int codPF;
					int codPj;

Mais claro

					int codigoPessoaJuridica;

Classes

  • Devem ser pequenas
  • Ter uma única responsabilidade (SRP)
  • Priorizar o encapsulamento
  • Nome de classes normalmente devem ser representadas por substantivos

Single Responsability Principle (SRP)

O princípio da responsabilidade única afirma que uma classe deve ter uma única responsabilidade, ou seja, apenas uma razão para mudar, e esta responsabilidade deve ser totalmente encapsulada pela classe.
Reduza o acoplamento, se o métodos faz duas ou três coisas diferentes ao mesmo tempo, deve ser considerado a divisão da funcionalidade.

Métodos

  • Devem ser pequenos
  • Ter uma única responsabilidade (SRP)
  • Os nomes dos métodos devem ser verbos
  • Evite repetição (DRY)
  • O nivel de identação não deve ser maior que 2
  • Poucos parâmetros

Para

public void save(String name, String cpf, String age)

Faça

public void save(Person person)

Comentários

  • Um código bem explicado não necessita de comentários
  • Se tiver que explicar o seu código, refatore-o
  • Comentários não são atualizados, e por isso mentem
					// verifica se conta está usando cheque especial
					if ( conta.getTipo().equals(Tipo.CORRENTE) && conta.getSaldo() < 0 ) {
						...
					}

Mais claro e descritivo

					if ( conta.estaUsandoChequeEspecial() ) {
						...
					}

Formatação

  • Classes menores são mais facil de entender
  • Conceitos relacionados devem ficar próximos
  • Limite o tamanho da linha (120 caracters - Uncle Bob)
  • Combine um estilo com a equipe (checkStyle Java - Maven)

Tratamento de Erros

  • Sempre lance exceções ao retornar um erro para o usuário
  • Forneça exceções de contexto
  • Não retorne null
  • Não passe null

Teste!

O teste é tão importante quanto o código de produção

Construção, operação e verificação

F.I.R.S.T.

Princípios de um teste unitário (F.I.R.S.T)
  • Fast - Um teste unitário deve ser executado rápido
  • Indepentent - Não deve depender de outro, podendo rodar em qualquer ordem
  • Repeatable - Pode ser executado quantas vezes quiser, mas trazendo o mesmo resultado
  • Self-Verifying - Auto verificavel, saber quando a execução foi correta e quando falhou
  • Timely - Deve ser criado no momento correto

Enfim

Não escreva código complexo, tempo e recursos gastos em um código ilegível serão muito maiores do que os obtidos com a otimização.

E mais importante, o code review deve ser motivacional e com compartilhamento de conhecimento, em vez de questinar a compêtencia
